# ChatGFT Admin: руководство по страницам

Документ описывает, где что находится в админке `chat-gft-admin`, за что отвечает каждая важная страница и как ей пользоваться.

---

## 1) Карта навигации (верхнее меню)

Основные разделы в `Navbar`:

- `Dashboard` — общая сводка.
- `Users` — пользователи и их данные.
- `Quests` — квесты/задания.
- `Payments` — платежи и инвойсы.
- `Products` — пакеты энергии/продукты.
- `Rewards` — награды (витрина).
- `Cases` — кейсы и содержимое кейсов.
- `Withdrawals` — запросы на вывод.
- `Daily Rewards` — ежедневные награды (7-дневный цикл).
- `Agents` — истории/агенты, миссии, stage rewards, step rewards.
- `Artifacts` — каталог артефактов.
- `LLM Debug` — трассы пайплайна LLM.
- `Трафик` — источники трафика, startapp-коды и аналитика.

---

## 2) Самая важная страница: Agents (подробно)

Раздел `Agents` — центральная точка управления историями и миссиями.

На странице есть 4 крупных блока:

1. **Agents** (список и CRUD самих агентов/историй)  
2. **Missions** (внутри модалки редактирования агента)  
3. **Stage Rewards** (награда за завершение этапа/миссии)  
4. **Step Rewards (Missions 1 & 2)** (награда за ключевые шаги внутри ранних миссий)

В шапке страницы **Agents** есть кнопка **«Выгрузить данные»** — JSON-бэкап агентов, миссий, компаньонов и наград (см. [§2.10](#210-выгрузка-данных-json)).

---

### 2.1 Агенты (истории): что это

Агент = конкретная история (например, `starwars`) со своим:

- `historyName` (технический уникальный ключ),
- display-именем (RU/EN),
- описанием (RU/EN),
- `systemPrompt`,
- порядком показа (`orderIndex`),
- статусом активности (`isActive`),
- медиа: avatar / preview / background / video.

---

### 2.2 Поля формы агента (AgentFormModal)

#### `History Name` (обязательно)

- Уникальный идентификатор истории.
- Используется как системный ключ (привязка чатов, статусов, прогресса).
- Рекомендация: использовать короткий стабильный slug, без пробелов, например: `starwars`.

#### `Display name (RU/default)` и `Display name (EN)`

- Название истории, которое видит пользователь.
- EN — опциональный перевод.

#### `Description (RU/default)` и `Description (EN)`

- Краткое описание истории для UI выбора истории.
- EN — опциональный перевод.

#### `Order Index`

- Порядок в списках (меньше = выше/раньше).

#### `Active`

- Включает/отключает историю для использования.

#### `System Prompt` (обязательно)

- Главная инструкция поведения агента.
- Влияет на стиль, тон и структуру ответа LLM.
- На форме показывается статистика (символы/слова).

---

### 2.3 Медиа агента (только при редактировании существующего агента)

Доступны отдельные загрузки/удаления:

- **Agent Avatar**
- **Agent Preview** (image/video/json)
- **Agent Background** (image/video/json)
- **Agent Video**

Практика:

- Перед заменой ассета убедитесь, что новый файл корректно открывается локально.
- После загрузки проверяйте превью в UI и поведение на клиенте.

### 2.4 Missions внутри агента (самый критичный блок)

Открывается внизу модалки агента (только в режиме редактирования агента).

#### Что можно делать

- Создавать миссию.
- Редактировать миссию.
- Удалять миссию.
- Загружать/удалять видео миссии.
- Привязывать артефакты к миссии.

#### Поля миссии

##### `Title`, `Title (EN)`
- Название миссии в RU/EN.

##### `Description`, `Description (EN)`
- Короткое описание миссии в RU/EN.

##### `UI step goals (chat)` — ключевое новое поле

- Текстовый ввод для целей шагов в чате (нижний блок “Цель” на клиенте).
- Формат: **один шаг на строку**.
- Рекомендуемый вид:

```text
1) Осмотреться и выбрать план
2) Добыть первичную наводку
3) Получить действенные детали
```

- Индексы должны совпадать с `main_step` из LLM (1-based).
- На сервер уходит строка `uiStepGoalsText`.
- Сервер парсит строку и сохраняет JSON `uiStepGoals` в миссии (в БД).

##### `Mission Prompt (LLM)`

- Детальный сценарный промпт миссии.
- Может содержать служебные теги для артефактов (через mention/helper).
- При **сохранении** миссии сервер парсит теги `@artifact(...)` из блока `Required beats` и записывает их в БД (`mission_artifacts`, `mission_artifact_steps`). Подробнее — [§2.4.1](#241-парсинг-artifact-при-сохранении-миссии).

#### Подробно: что конкретно должно быть в `Mission Prompt (LLM)`

`Mission Prompt` должен быть не просто художественным текстом, а четкой спецификацией поведения LLM в миссии.
Рекомендуемая структура:

1. `## Mission goal`
2. `## Required beats (linear core)`
3. `## Key locations / NPC`
4. `## StoryState bootstrap (JSON)`

##### 1) `Mission goal`: обязательные три пункта

- `Objective` — что именно нужно добыть/сделать.
- `Success criteria` — по какому факту миссия считается завершенной.
- `Hard constraints` — ограничения, которые не дают “проскочить” сюжет слишком быстро.

Пример паттерна:

- `Objective`: Добыть X.
- `Success criteria`: Герой получил X и вышел из зоны Y.
- `Hard constraints`: Нельзя решить всё одним действием; прямой штурм повышает риск тревоги.

##### 2) `Required beats (linear core)`: ключевая часть

**Критично:** заголовок секции должен начинаться с markdown-заголовка **`## Required beats`** (регистр не важен).  
Вариант `Required beats (КЛЮЧЕВЫЕ ШАГИ)` **без** `##` парсер **не увидит** — теги `@artifact(...)` не попадут в БД.

```text
## Required beats (КЛЮЧЕВЫЕ ШАГИ)

1) ...
2) ...
```

Оптимально 5 шагов (допустимо 4–6), где каждый шаг построен по формуле:

**Контекст шага → цель шага → чёткое условие завершения → `@progress(flag)`**

Рекомендуемый шаблон строки:

```text
N) [Контекст]. Герою нужно [конкретная цель]. Шаг завершается, когда [проверяемый факт]. @progress(flag_name)
```

Если на шаге должен быть артефакт через side-цепочку:

```text
... @progress(flag_name) @artifact(ARTIFACT_CODE) @artifact_side(2)
```

Поддерживаемые теги в beats:

| Тег | Пример | Куда сохраняется |
|-----|--------|------------------|
| `@progress(flag)` | `@progress(info_found)` | только в тексте промпта + `mission_flags` в bootstrap JSON |
| `@artifact(CODE)` | `@artifact(FAKE_ID)` | **БД:** `mission_artifacts` + `mission_artifact_steps` |
| `@artifact(CODE,USE,1)` | `@artifact(BLASTER,USE,1)` | то же, action `USE`, amount `1` |
| `@artifact_side(K)` | `@artifact_side(2)` | **только текст промпта** (runtime читает из промпта) |
| `@artifact_detour(K)` | устаревший синоним | парсится как `@artifact_side`, в БД не пишется |

Код артефакта в `@artifact(...)` должен **точно совпадать** с `code` в разделе **Artifacts**. Если кода нет в каталоге — тег при save **пропускается** (в логах: `Skipped tag: artifact not found in DB`).

###### Какие паттерны уже хорошо используются в ваших примерах

- Линейная арка из 5 вех: разведка -> доступ -> ключевое действие -> осложнение -> выход.
- У большинства шагов есть четкое “шаг завершен, когда...”.
- Артефакты внедрены как detour, не ломая core-поток.
- Ограничения миссии вынесены отдельно в `Hard constraints`.
- Есть привязка к локациям/NPC, что помогает LLM давать правдоподобные варианты.

###### Правила качества для шагов

- Один шаг = одна проверяемая веха (не смешивать 2-3 большие цели в один beat).
- Формулировать завершение как факт состояния, а не намерение.
  - Хорошо: “герой уже знает окно времени и код коридора”.
  - Плохо: “герой пытается узнать детали”.
- В каждом шаге должен быть только один `@progress(...)`, и он должен быть уникален в миссии.
- Для специальных механик (например “на 3-й попытке detour становится core”) правило нужно писать явно в нужном шаге.

##### 3) `Key locations / NPC`

Краткий список мест и персонажей, через которые LLM может строить сцены и предложения действий.
Достаточно 2-5 локаций и 2-4 NPC, связанных с целями миссии.

##### 4) `StoryState bootstrap (JSON)`

Содержит `mission_flags` инициализированные в `false`.
Ключевое требование: флаги в JSON должны совпадать с тегами `@progress(...)` в шагах.

Пример корректного соответствия:

```text
@progress(route_known)
```

```json
{
  "mission_flags": {
    "route_known": false
  }
}
```

###### Важная проверка на консистентность (по вашим примерам)

- Убедитесь, что каждый `@progress(...)` присутствует в `mission_flags`.
- Имена должны совпадать 1:1 (без синонимов и расхождений вроде `vault_found` vs `cargo_found`).
- Неиспользуемые флаги лучше удалять, чтобы не размывать логику.

##### Мини-чеклист перед сохранением Mission Prompt

- [ ] Есть все 4 секции (`## Mission goal`, `## Required beats`, `## Key locations / NPC`, `## StoryState bootstrap`).
- [ ] Заголовок beats — **`## Required beats`**, шаги нумерованы как `N) текст`.
- [ ] В каждом beat есть понятный критерий завершения.
- [ ] В каждом beat есть уникальный `@progress(flag)`.
- [ ] Все `@progress`-флаги совпадают с `mission_flags` в bootstrap JSON.
- [ ] Артефакты на нужных шагах: `@artifact(CODE)` и код есть в каталоге **Artifacts**.
- [ ] Порядок шагов логически ведёт к `Success criteria`.

##### 2.4.1 Парсинг `@artifact` при сохранении миссии

При **Create/Update** миссии backend (`agentController`) делает следующее:

1. Ищет секцию **`## Required beats`** в `missionPrompt`.
2. Внутри неё — строки вида `N) ...` (номер шага + текст).
3. Из текста шага извлекает `@artifact(CODE)` или `@artifact(CODE,RECEIVE|USE,amount)`.
4. Для каждого найденного кода:
   - ищет артефакт в таблице `artifacts` по полю `code`;
   - создаёт/обновляет запись в **`mission_artifacts`** (связь миссия ↔ артефакт);
   - создаёт запись в **`mission_artifact_steps`** (шаг, action, amount, `flagKey`).

При **Update**, если в body передан `missionPrompt`, старые `mission_artifact_steps` для миссии **удаляются** и пересобираются из промпта заново.

**Не парсятся автоматически в БД:**

- `@artifact_side(K)` / `@artifact_detour(K)` — остаются только в тексте промпта;
- теги вне блока `## Required beats` (например в `Mission goal` без заголовка beats);
- `@artifact(SKRI_COMPANION)` и прочие коды, которых нет в каталоге артефактов (компаньоны — отдельная сущность).

**Чекбоксы «Artifacts in this mission»** — отдельный путь: явный список `artifactIds` в API. Его можно комбинировать с тегами в промпте; при update с `artifactIds` список в `mission_artifacts` перезаписывается из чекбоксов, а step-записи — из промпта.

**Как проверить, что привязка сохранилась:**

1. Сохранить миссию → открыть снова Edit (в ответе API у миссии должны быть `artifacts` / `artifactSteps`).
2. Нажать **«Выгрузить данные»** на странице Agents → в JSON у миссии непустые `artifactCodes` и `artifactSteps`.
3. Или SQL: `SELECT * FROM mission_artifact_steps WHERE "missionId" = ...`.

##### `Artifacts in this mission`

- Чекбоксы артефактов (из каталога `Artifacts`).
- Это список “какие артефакты вообще могут быть в миссии”.

##### `Order Index`

- Порядок миссии внутри агента.

### 2.5 Как правильно заполнять `UI step goals (chat)`

#### Правила

- Только нумерованные строки.
- Нумерация начинается с `1`.
- Один номер = один ключевой шаг.
- Текст должен быть коротким, ориентированным на действие игрока.
- Без тэгов `@progress`, `@artifact` и т.д. (они нужны в prompt, но не в UI goals).

#### Хорошо

```text
1) Осмотреться и выбрать способ добычи данных о курьере
2) Получить первичную подтвержденную наводку
3) Уточнить маршрут и окно времени курьера
4) Перехватить данные или ключ-часть
5) Уйти из зоны риска с добытыми данными
```

#### Плохо

```text
Осмотреться
Дальше действовать
```

(нет индексов, не будет корректного парсинга)

### 2.6 Типовой рабочий сценарий: “создать/обновить миссию”

1. Открыть `Agents`.
2. Выбрать нужного агента → `Edit`.
3. В блоке `Missions` нажать `Create Mission` или `Edit`.
4. Заполнить `Title`, `Mission Prompt`, `UI step goals (chat)`, `Order Index`.
5. При необходимости отметить артефакты чекбоксами.
6. Нажать `Create/Update`.
7. Проверить:
   - миссия сохранилась в списке,
   - порядок верный,
   - при повторном открытии `Edit` текст `UI step goals` подставился обратно,
   - если в промпте есть `@artifact(...)` — после save в export JSON у миссии заполнены `artifactCodes` / `artifactSteps`.

### 2.7 Диагностика проблем в Agents/Missions

#### Проблема: “UI step goals не сохраняется”

Проверьте:

1. В Network у запроса `PUT /api/agent/:id/missions/:missionId` реально есть `uiStepGoalsText`.
2. Формат строк корректен (`1) ...`, `2) ...`).
3. На бэкенде применена колонка `uiStepGoals` в таблице `missions`.
4. Задеплоена актуальная версия `agentController`/`models`.
5. Логи `Mission UPDATE` (добавлены в контроллер):
   - `incoming` (есть ли ключ в body),
   - `uiStepGoals parse`,
   - `after mission.save()`,
   - `reload from DB for response`.

#### Проблема: “миссия не на том месте”

- Проверить `Order Index`.
- Убедиться, что нет дублирующего `orderIndex` для того же агента.

#### Проблема: «нет артефакта в выборе» (чекбоксы)

- Артефакт должен существовать в разделе `Artifacts`.
- Обновите страницу/переоткройте форму миссии (список артефактов подгружается из store).

#### Проблема: «в промпте есть `@artifact`, но в export пустые `artifactCodes`»

Типичные причины:

1. **Нет заголовка `## Required beats`** — только `Required beats` без `##`. Исправить промпт и пересохранить миссию.
2. **Шаг не в формате `N) ...`** — парсер не распознаёт beat.
3. **Код артефакта не существует** в `Artifacts` (опечатка или артефакт не создан).
4. Смотрели старый export до пересохранения миссии.

Export **не парсит промпт заново** — он отдаёт только то, что уже лежит в `mission_artifacts` / `mission_artifact_steps`.

#### Проблема: «`@artifact_side` не виден в export»

Это нормально: side-теги **не хранятся в отдельной таблице**, только в поле `missionPrompt`. В JSON export они остаются внутри текста промпта.

### 2.8 Stage Rewards (в Agents)

Блок `Stage Rewards` — награды за завершение этапа (миссии).

Что настраивается:

- `stageNumber`
- `rewardAmount`
- опционально `rewardCaseId`
- `isActive`

Использование:

- Create/Edit/Delete по stage number.
- Включение/отключение награды без удаления через `isActive`.

---

### 2.9 Step Rewards (Missions 1 & 2)

Блок `Step Rewards (Missions 1 & 2)` — точечные награды за правильный шаг.

Поля:

- `missionOrderIndex`
- `stepNumber`
- `rewardGems`

Сценарий:

- Настраивайте только шаги, за которые реально хотите выдавать gems.
- Следите за консистентностью с тем, как LLM увеличивает `main_step`.

### 2.10 Выгрузка данных (JSON)

На страницах **Agents**, **Artifacts**, **Cases** и **Rewards** в шапке есть кнопка **«Выгрузить данные»** (вторая кнопка слева от основного действия). Скачивается JSON-файл с бэкапом конфигурации из БД.

#### Agents → `agents_export_YYYY-MM-DD.json`

API: `GET /api/agent/export` (только ADMIN).

Содержимое:

| Поле | Описание |
|------|----------|
| `schemaVersion` | версия формата (сейчас `1`) |
| `exportedAt` | ISO-дата выгрузки |
| `agents[]` | агенты: поля истории, `videoId` / `avatarId` / … (без объектов медиа) |
| `agents[].companion` | компаньон истории или `null` |
| `agents[].missions[]` | миссии: тексты, `uiStepGoals`, `videoId`, `artifactCodes`, `artifactSteps` |
| `stageRewards[]` | награды за этапы (блок на странице Agents) |
| `missionStepRewards[]` | награды за шаги миссий 1 и 2 |

**Не включается:** бинарные файлы MinIO, объекты `media_files`, пользовательский прогресс.

Связи артефактов в миссиях — по **`artifactCode`**, не по numeric id (удобно для переноса между окружениями).

#### Artifacts → `artifacts_export_YYYY-MM-DD.json`

API: `GET /api/artifact/export` (только ADMIN).

Каждый артефакт: `code`, названия, описания, `level`, `boostType`, `buyPrice`, `sellPrice`, `mediaId`, `agentHistoryName`. Медиа-объекты не включаются.

#### Cases → `cases_export_YYYY-MM-DD.json`

API: `GET /api/case/export` (только ADMIN).

Каждый кейс: `name`, `nameEn`, `description`, `descriptionEn`, `price`, `image`, `isActive`, массив `items[]`.

Элементы кейса:

| `type` | Поля в export |
|--------|----------------|
| `gems` / `energy` | `amount`, `weight` |
| `reward` | `rewardName`, `weight` |

**Не включается:** объекты `media_files`, история открытий (`user_cases`), numeric `rewardId`.

Связь с наградами — по **`rewardName`**, не по id (удобно для переноса между окружениями; награда должна существовать в целевой БД с тем же именем).

#### Rewards → `rewards_export_YYYY-MM-DD.json`

API: `GET /api/reward/export` (только ADMIN).

Каждая награда: `name`, `price`, `tonPrice`, `description`, `isActive`, `onlyCase`, `previewId`.

**Не включается:** numeric `id`, объекты `mediaFile` / `preview`, основная анимация (Lottie в `media_files` по `entityType: reward`).

Связь с кейсами — по **`name`** (должно совпадать с `rewardName` в `cases_export`).

#### Зачем использовать

- бэкап перед большими правками;
- перенос конфигурации test → prod (вместе с MinIO по `mediaId` / `videoId`);
- сверка: после save миссии проверить, что `artifactCodes` в export не пустые.

#### Полный бэкап истории starwars

Нужны **оба** файла + файлы в MinIO:

1. `agents_export_*.json`
2. `artifacts_export_*.json`
3. синхронизация bucket MinIO (по id из export)

## 3) Важные страницы (краткий практический справочник)

Ниже — что важно знать по остальным ключевым разделам.

### Dashboard
- Сводный экран состояния системы.
- Используется для быстрого контроля здоровья проекта.

### Users
- Поиск и просмотр пользователей.
- Переход в `User Details` для углубленной диагностики профиля и активности.

### User Details
- Детальная карточка пользователя: идентификаторы, профиль, связанная активность.
- **Журнал артефактов** — все изменения инвентаря: `RECEIVE`/`USE` в чате, `BUY`/`SELL` на маркете, `BURN_LEVEL` при открытии уровня, `FIRST_MISSION_GRANT`, `ADMIN_GRANT`. Сводка gems — только по BUY/SELL.
- Используйте для поддержки и разбора точечных инцидентов.

### Quests
- Управление заданиями и квестами.
- Изменения тут влияют на получение энергии/гемов и вовлечение.

### Payments
- Работа с заказами/инвойсами.
- Используется для контроля оплаты и финансовых операций.

### Products
- Пакеты энергии и цены.
- Проверяйте корректность цен, названий, активности и отображения в клиенте.

### Rewards
- Управление наградами, визуалами и связанными параметрами выдачи.
- Кнопка **«Выгрузить данные»** — JSON-бэкап каталога (см. [§2.10](#210-выгрузка-данных-json)).

### Cases
- Кейсы и их содержимое (состав, вероятности, визуал).
- Важно для экономики и предсказуемости выдач.
- Кнопка **«Выгрузить данные»** — JSON-бэкап кейсов (см. [§2.10](#210-выгрузка-данных-json)); `rewardName` в items ссылается на `name` из `rewards_export`.

### Withdrawals
- Очередь запросов на вывод призов.
- Операционная страница: обработка статусов и коммуникация.

### Daily Rewards
- Настройка ежедневных наград (обычно 7 дней).
- Следите за балансом экономики (не завышать выдачу).

### Artifacts
- Каталог артефактов: коды, названия, описания, медиа.
- Основа для миссий и artifact-действий в сценариях.
- Кнопка **«Выгрузить данные»** — JSON-бэкап каталога (см. [§2.10](#210-выгрузка-данных-json)).
- Код артефакта (`code`) должен совпадать с аргументом `@artifact(CODE)` в промпте миссии.

### LLM Debug
- Трассы полного пайплайна: вход/выход, дебаг-поля, вычисления.
- Первая страница для расследования “почему LLM/шаг/подсказка сработали не так”.

### Трафик
- Источники трафика и startapp-коды.
- Аналитика по поведению/покупкам привлеченных пользователей.

## 4) Мини-чеклист перед релизом изменений в админке

- [ ] Поля формы сохраняются и повторно читаются без потерь.
- [ ] Валидация работает (required/формат/уникальность).
- [ ] Новые поля видны в ответах API и в UI.
- [ ] Для новых сценариев есть запись в этом документе.
- [ ] Проверен happy path + минимум один негативный кейс.
